# 房间构建与风水系统商业级开发计划

> 本文用于规划 当前生产主线的“房间构建、房间识别、风水计算、建造交互、持久化恢复与运维验证”系统。
>
> 本文是开发计划，不表示当前主线已经完成。主工作区固定为 `packages/*`，不恢复旧线目录作为新实现入口。

---

## 目标结论

房间与风水系统不能做成“按建筑类型写死规则”的玩法脚本。商业级 MMO 的正确形态是：

```text
无限扩展的建筑/家具配置
-> 启动期校验并编译为 trait / topology / numeric vector
-> 服务端权威建造命令写入 BuildingInstance
-> BuildingInstance 投影到 RuntimeTilePlane 与拓扑索引
-> 拓扑变化触发增量房间识别
-> 房间聚合快照驱动风水规则引擎
-> 风水 snapshot 持久化、可解释、可重算
-> 客户端只做预览、overlay、详情展示和操作意图
```

核心原则：

- 建筑、家具、墙体、门窗、装饰、设施类型可以无限扩展。
- 房间判定不认识具体类型，只认识预编译拓扑能力。
- 风水计算不认识具体类型，只读取房间聚合指标、trait、五行向量和规则表。
- 高频路径不解析 JSON、不查 schema、不拼字符串坐标、不做全图扫描。
- `BuildingInstance` 是长期真源；`RoomInstance` 和 `FengShuiSnapshot` 是可持久化、可校验、可重算的派生快照。
- 网络同步按 AOI、按需详情、低频 patch 分层，不污染高频 world delta。
- 所有会影响材料、地图、占位、房间收益的状态变化必须由服务端受控流程裁定。

第一版验收闭环：

```text
配置新增建筑/家具
-> 服务端启动期编译为能力索引
-> 玩家建造墙/门/窗/家具
-> 服务端扣材料并更新实例拓扑
-> 自动识别房间
-> 自动生成风水 snapshot 与 reasons
-> 客户端按需查看房间与风水
-> 重启后建筑、房间、风水可恢复或自动重算
```

## 架构原则

### 类型无限扩展，运行时语义固定

禁止在运行时这样写：

```ts
if (defId === 'alchemy_furnace') {
  score += 30;
}
```

运行时只能读取稳定语义：

```ts
if (aggregate.traitCounts.get('facility.alchemy.heat_source') > 0) {
  applyRule('alchemy_heat_source');
}
```

建筑配置可以无限扩展，但必须编译为固定口径：

- `topology`：是否阻挡移动、是否阻挡视线、是否形成房间边界、是否门窗开口、是否遮蔽屋顶、是否挡煞。
- `placement`：层级、footprint、权限、地形约束、是否可重叠。
- `fengShui`：五行向量、trait 列表、舒适、稳定、聚气、泄气、煞气、完整性影响。
- `economy`：材料消耗、返还、建造时间、拆除规则。
- `interaction`：是否可交互、交互动作、详情查询能力。
- `visual`：tileType、glyph、颜色、客户端展示 catalog。

### 房间判定只认 topology

房间识别不能关心“这是玄铁墙还是木墙”。它只关心：

- 当前 cell 是否可作为室内流动空间。
- 边界方向是否被 `roomBoundary` 阻隔。
- 边界上是否存在 `opening = door/window`。
- 是否有足够 `roofCoverage`。
- 是否通过开口或空洞触达地图外开放空间。
- 是否有 `semiOutdoorLink` 构成庭院、回廊、半室外。

### 风水计算只认 RoomAggregate

风水不能遍历每个家具后判断类型。它读取房间聚合快照：

- 面积、周长、门窗数量、封闭性、屋顶覆盖。
- 五行向量。
- trait 计数。
- 灵气密度、煞气暴露、舒适度、稳定度、完整性。
- 阵法覆盖、资源覆盖、破损惩罚。

建筑/家具新增时，只需要新增配置；规则新增时，只新增规则表。

### 热路径只读预编译结构

启动期或配置发布期完成：

- JSON/schema 校验。
- trait 字符串注册为数值 ID。
- 五行转定长数字向量。
- footprint 旋转预计算。
- topology mask 预计算。
- 风水规则编译为受限条件树。

运行时不得：

- [x] 解析编辑器草稿。
- [x] 逐条查 schema。
- [x] 按字符串 defId 分支。
- [x] 对每次计算拼 `"x,y"`。
- [x] 在 tick 热路径 `JSON.stringify` / `JSON.parse`。
- [ ] 因单个房间变化扫描全图或全实例建筑。

## 房间识别算法

### 触发条件

- [ ] 新建、拆除、升级、损坏、修复 `topology.roomBoundary > 0` 的建筑。
- [ ] 门、窗、回廊、屋顶覆盖等拓扑能力变化。
- [ ] 地图动态扩展或激活/失活地块。
- [ ] GM 强制重算。
- [ ] 配置 revision 变化后需要重建拓扑索引。

普通家具只影响 `RoomAggregate`，不触发房间识别。

### 识别流程

```text
建造/拆除命令
-> 更新 BuildingInstance
-> 更新 CellTopologyArrays
-> 标记 dirty rect
-> 从 dirty rect 收集旧 roomId 与候选可走 cell
-> 清理受影响 room 的 roomIdByCell
-> 对候选 cell 做预算化 flood fill
-> 生成或更新 RoomInstance
-> 生成 roomHash
-> hash 变化才标记 fengShui dirty
```

关键规则：

- dirty rect 只用于找候选，不能把 flood fill 限死在 dirty rect 内。
- flood fill 可以跨出 dirty rect，但必须有 cell budget。
- 超预算任务进入 deferred queue，分 tick 继续处理。
- 触达不存在格子、地图开放边界或未封闭边界时，判为开放空间。
- 门是 opening，不是完整墙；窗是否开口由 `openingKind` 和 `blocksMove/blocksSight` 共同决定。
- 回廊、庭院、半室外通过 `semiOutdoorLink` 和 `roofCoverageRatio` 判定。

### 性能约束

- [ ] 单个 1x1 建筑导致的 topology 更新 p95 小于 1ms。
- [ ] 20x20 房间识别 p95 小于 8ms。
- [x] 单次 flood 默认预算不超过 4096 cell。
- [x] 大型宗门/洞府房间识别记录 deferred 起点，不阻塞地图 tick。
- [x] `roomIdByCell` 使用 `Int32Array` 或等价数字数组，不使用字符串坐标 map 作为热路径。
- [x] 房间识别不得扫描全实例所有建筑，除非 GM 显式全量重建。

## 风水计算架构

### 配置驱动规则

风水规则使用受限 DSL，不允许执行任意 JS。

```ts
interface FengShuiRule {
  id: string;
  priority: number;
  when: FengShuiCondition[];
  scoreDelta: number;
  capGroup?: string;
  reasonCode: string;
  severity: 'info' | 'good' | 'warning' | 'bad';
}
```

条件只能来自白名单：

- `roomRoleIs`
- `enclosedIs`
- `metricGte / metricLte`
- `traitAtLeast / traitMissing`
- `elementDominates`
- `elementGeneratesFunction`
- `elementConflictsFunction`
- `qiDensityBand`
- `shaExposureBand`
- `formationCoverageBand`
- `integrityBand`

运行期输入固定：

```text
RoomInstance + RoomAggregate + compiled FengShuiRuleTable -> FengShuiSnapshot
```

### 增量聚合

不同事件走不同代价：

| 事件 | 房间识别 | 聚合更新 | 风水重算 |
|---|---:|---:|---:|
| 建墙/拆墙 | 是 | 是 | 是 |
| 加门/窗 | 是 | 是 | 是 |
| 放床/蒲团/丹炉 | 否 | delta | 是 |
| 家具破损/修复 | 否 | delta | 是 |
| tile damage 变化 | 可能 | delta | 是 |
| 灵气资源变化 | 否 | 批处理 | 是 |
| 阵法覆盖变化 | 否 | delta | 是 |
| 玩家移动 | 否 | 否 | 否 |
| 普通战斗 tick | 否 | 否 | 否 |

### 风水评分

首版建议固定总分 `0..1000`，但每个维度由配置权重控制：

```text
score =
  base
+ enclosureScore
+ shapeScore
+ qiScore
+ elementScore
+ comfortScore
+ formationScore
- shaScore
- integrityPenalty
- conflictPenalty
```

所有原因必须输出 `reasonCode`，客户端本地化展示，不同步长文本。

### 不能做的事情

- [x] 不能每 tick 主动修改玩家属性。
- [x] 不能每 tick 重算所有房间。
- [x] 不能为了客户端展示返回完整规则表。
- [x] 不能用建筑名、家具名、defId 写死评分。
- [x] 不能把运营可编辑规则变成任意脚本执行。

## 持久化设计

### 真源边界

| 状态 | 真源 | 说明 |
|---|---|---|
| BuildingInstance | PostgreSQL 分域表 | 长期真源，材料、归属、拆除、审计都依赖它 |
| CellTopologyArrays | 运行时派生 | 由 BuildingInstance + tilePlane 重建 |
| RoomInstance | 可持久化派生快照 | 加速恢复，但 hash 不一致必须重算 |
| RoomAggregate | 运行时派生 | 首版不作为长期真源，后续可加缓存表 |
| FengShuiSnapshot | 可持久化派生快照 | 展示和玩法结算读取，revision 不一致可重算 |
| 风水规则 | 内容配置 | 启动期编译，运行时不解析草稿 |

### 推荐分域表

- [x] `instance_building_state`
- [x] `instance_building_cell`
- [x] `instance_room_state`
- [x] `instance_room_cell`，首版可用压缩 payload 或 chunk，后续再拆行级表。
- [x] `instance_fengshui_state`
- [x] `instance_building_audit_log`
- [x] `instance_building_operation_idempotency`

### 恢复流程

```text
load instance tile cells
-> load building state
-> rebuild building cell occupancy
-> rebuild topology arrays
-> load room/fengShui snapshot
-> 校验 roomHash/topologyRevision/contentRevision
-> 不一致则标记 deferred rebuild
-> 服务可先启动，受影响房间暂显“计算中”
```

### 幂等与资产安全

- [x] `requestId` 和 playerId 组合做运行时建造/拆除幂等。
- [ ] 材料扣除和 BuildingInstance 创建必须在同一受控操作链内。
- [x] 失败不得出现扣材料但无建筑。
- [x] 重复请求不得重复扣材料。
- [ ] 拆除返还必须以服务端建筑状态为准。
- [ ] GM 修复不得绕过审计直接制造资产变化。

## 风险与规避

| 风险 | 表现 | 规避 |
|---|---|---|
| 类型扩展导致算法改不动 | 每加一个家具都改代码 | 配置 trait + topology + numeric vector，算法不看 defId |
| 房间识别卡 tick | 大地图 flood fill 阻塞 | dirty candidate + budget + deferred queue |
| 风水计算写成脚本引擎 | 性能不可控、不可审计 | 受限 DSL + 启动期编译 |
| 持久化写放大 | 每次重算写大量 room tile | 首版控制 room cell 持久化粒度，后续 row-level delta |
| 网络包体膨胀 | 高频包带完整风水详情 | detail query + overlay 按需 + reasonCode |
| 客户端作弊 | 客户端伪造建造/材料 | 服务端权威校验和 requestId 幂等 |
| 恢复不一致 | 建筑、房间、风水互相漂移 | BuildingInstance 为真源，room/fengShui hash 校验可重算 |
| UI 操作被刷新打断 | 打开面板后频繁重建 | MapStore overlay + 面板局部 patch |
| 规则不可解释 | 玩家只看到玄学分数 | reasonCode + 指标详情 + 改造建议 |
